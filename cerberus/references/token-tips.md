# Spending fewer tokens in the pipeline

Cerberus runs three subagents per task, so cost is a first-class concern. These tactics cut token spend without gutting the plan → implement → review structure that makes the pipeline worth running. Pull from here when the user asks to reduce cost or when a weekly rollup (`cost-tracking.md`) looks high.

The single biggest lever is **cache reads**: they usually dominate the token count but are priced at ~1/10th the input rate. Most "expensive" sessions are expensive because of *uncached input* (fresh exploration) and *output* (long generations), not cache reads. Optimize those two first.

## 1. Right-size the pipeline — skip heads that don't earn their cost

Not every task needs all three heads. Trimming is the largest single saving because it removes a whole subagent.

- **Trivial / mechanical change** (one-line fix, rename, obvious typo) → skip planning. Implement, then a light review, or just do it inline in the orchestrator and say so.
- **Docs / comments / config with no runtime surface** → planning and review add little. A single implementation pass often suffices.
- **"How does X work?" / exploration** → planning-only. Dispatch head 1, return its findings, stop. There may be nothing to build yet.
- **A stuck bounce loop** → after 2–3 bounces without convergence, stop looping and hand the reviewer's notes to the user. More passes just burn tokens.

Always tell the user when you trim a head, so the shorter pipeline is a visible choice, not a silent gap.

## 2. Scope each subagent's context tightly

Uncached input is the expensive kind, and it's mostly generated by *exploration* — a subagent grepping and reading its way around an unfamiliar tree. Cut it by handing the subagent its context instead of making it hunt:

- **Name the specific files** the task touches in the dispatch prompt. A planner told "look at `AuthService.swift` and `LoginView.swift`" reads two files; one told "fix the login bug" reads twenty.
- **Pass the plan verbatim into implementation**, so the implementer doesn't re-derive it.
- **Pass the diff/commit hash and the implementer's summary into review**, so the reviewer reads the change directly instead of reconstructing what happened.
- **Don't attach the whole repo.** Precision beats breadth; a subagent rarely needs more than the handful of files in scope plus `CLAUDE.md`.

## 3. Exploit the prompt cache

Cache reads cost ~10× less than fresh input. Structure each subagent's context so the stable parts sit at the front and get reused:

- **Put the invariant preamble first** — `CLAUDE.md`, conventions, the plan — before the task-specific asks. Stable prefixes are what the cache keys on.
- **Reuse the same framing across phases** so the shared preamble stays cache-warm rather than being re-sent as fresh input each time.
- Don't reorder or lightly edit that preamble between dispatches — a one-character change upstream invalidates the cached suffix.

## 4. Right-size the models

The default tiers (Fable → Sonnet → Opus) already match model strength to phase difficulty. Push further when cost matters:

- **Don't run Opus where Sonnet suffices.** Review of a small, low-risk change can go to a mid model — but never below the implementation tier (see `workflow.md`).
- **Don't plan on the top model for a one-file change.** If the approach is obvious, a mid-tier planner (or no planner) is enough.
- **Reserve the top model for genuinely reasoning-heavy planning** — cross-cutting changes, ambiguous root causes, real design decisions. That's where it pays for itself.

## 5. Reduce output tokens

Output is priced highest per token and is fully under your control:

- **Ask subagents to return concise summaries**, not full file dumps or restated diffs. The orchestrator needs the outcome, not a transcript.
- **Avoid make-work generations** — don't have a subagent reproduce a file it only changed three lines in, or narrate every step.
- **Batch verification.** Run the verify command once per phase after the change is complete, not after every micro-edit — each run's output is tokens the subagent reads back.

## 6. Coordinate cheaply

- **Refresh pricing once per day**, cached (`cost-tracking.md`) — never re-fetch per task.
- **Keep the orchestrator small.** Its whole value is staying out of the large plan/implement/review contexts. If you catch yourself doing the work in the orchestrator, you've lost the isolation *and* the cost benefit — dispatch instead.
- **Group related small tasks** so shared context (conventions, a common area of code) is read once by a subagent rather than re-read across several dispatches.

## Quick diagnosis when a rollup is high

1. **Which phase dominates?** Break the week down by phase. If review dwarfs everything, consider a lighter reviewer for low-risk changes. If planning dominates, you may be over-planning trivial tasks.
2. **Which token type dominates?** High *uncached input* → subagents are exploring too much; scope their context (tactic 2). High *output* → generations are too verbose (tactic 5). High *cache read* alone is usually fine — it's cheap.
3. **How many bounces?** Repeated implement→review bounces multiply cost. If a task bounced three times, the plan was probably weak — invest more in head 1 next time, which is cheaper than repeated head-2/head-3 passes.
